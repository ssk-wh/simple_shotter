## 1. 测试框架与环境

### 框架选型

| 组件 | 选型 | 说明 |
|------|------|------|
| 单元测试 | Qt Test (QTest) | 与 Qt 生态一致，支持信号槽测试 |
| GUI 测试 | Qt Test + QSignalSpy | 模拟鼠标/键盘事件 |
| 性能测试 | QElapsedTimer + QBENCHMARK | Qt 内置基准测试 |
| Mock | 手动 Mock（平台抽象层接口） | 通过 PlatformApi 接口注入测试实现 |

### 测试目录结构

```
tests/
├── CMakeLists.txt
├── test_screen_capture.cpp
├── test_window_detector.cpp
├── test_control_detector.cpp
├── test_region_selector.cpp
├── test_clipboard_manager.cpp
├── test_file_saver.cpp
├── test_hotkey_manager.cpp
├── test_integration.cpp
└── mock/
    ├── mock_platform_api.h
    └── mock_screen_provider.h
```

## 5. 性能测试

### 5.1 性能指标基线

| 指标 | 目标值 | 测量方法 | 优先级 |
|------|--------|----------|--------|
| 截图启动延迟（快捷键到覆盖层显示） | < 200ms | QElapsedTimer: 从信号触发到 overlay `showEvent` | P0 |
| 全屏截图耗时 | < 100ms | QBENCHMARK 包裹 `captureScreen()` | P0 |
| 窗口检测耗时 | < 100ms | QBENCHMARK 包裹 `detectWindows()` | P0 |
| 控件检测耗时 | < 200ms | QBENCHMARK 包裹 `detectControls()` | P1 |
| 鼠标移动响应（窗口高亮切换） | < 16ms (60fps) | 鼠标 moveEvent 到 paintEvent 完成 | P0 |
| 复制到剪贴板 | < 100ms | QBENCHMARK 包裹 clipboard 操作 | P1 |
| 保存文件 (PNG 1080p) | < 500ms | QBENCHMARK 包裹文件保存 | P1 |
| 内存基线（待机） | < 30MB | 应用启动后稳定时的 RSS | P1 |
| 内存峰值（截图中） | < 150MB | 截图模式活跃时的 RSS | P1 |

| PF-006 | 控件检测 (简单窗口) | 基准测试 | 检测耗时 |
| PF-007 | 控件检测 (复杂窗口 >100控件) | 基准测试 | 检测耗时 |
| PF-008 | 连续截图 20 次 | 循环执行 | 内存是否持续增长（泄漏检测） |
| PF-009 | 长时间待机 (1小时) | 启动后放置 | 内存是否持续增长 |
| PF-010 | 鼠标快速移动时的高亮响应 | 快速扫过多个窗口 | 高亮切换是否流畅、无闪烁 |

### 5.3 内存泄漏检测

- Windows: 使用 Visual Studio 内置 CRT 检测 (`_CrtSetDbgFlag`)
- Linux: 使用 Valgrind (`valgrind --leak-check=full`)
- 重点关注：
  - `QPixmap` 截图缓存的释放
  - 每次截图/取消循环后内存是否回到基线
  - 平台 API 句柄泄漏 (HWND、GC 等)

## 8. 回归测试策略

### 8.1 自动化测试级别

```
                     手动测试 (UX / 视觉 / 跨平台验证)
                    /
               集成测试 (关键流程 E2E)
              /
         功能测试 (各模块 API 级别)
        /
   单元测试 (纯逻辑、工具类)
```

| 级别 | 自动化程度 | 执行频率 | 覆盖内容 |
|------|-----------|----------|----------|
| 单元测试 | 100% 自动化 | 每次提交 | 纯逻辑：FileSaver文件名生成、RegionSelector区域计算 |
| 功能测试 | 90% 自动化 | 每次提交 | 各模块公共 API（需要 Mock 平台层） |
| 集成测试 | 70% 自动化 | 每日构建 | 关键截图流程 E2E |
| 手动测试 | 手动 | 版本发布前 | UX、视觉效果、跨平台验证 |

### 8.2 CI/CD 集成

```
git push
  |
  v
[Build] --> [Unit Tests] --> [Functional Tests] --> [Integration Tests]
  |              |                  |                       |
  Win + Linux    快速反馈(<1min)    Mock 平台层(<3min)      需要 GUI 环境
```

- **构建矩阵**: Windows (MSVC) + Linux (GCC)
- **测试触发**: 每次 push 和 PR 自动执行
- **GUI 测试**: 使用虚拟帧缓冲 (`xvfb-run` on Linux)

### 8.3 回归测试用例选择策略

| 变更类型 | 回归范围 | 说明 |
|----------|----------|------|
| core/ 目录变更 | 全量单元测试 + 集成测试 | 核心逻辑变更影响面广 |
| ui/ 目录变更 | UI 相关单元测试 + 集成测试 | 界面变更需要视觉验证 |
| platform/ 目录变更 | 对应平台全量测试 | 平台实现变更需全面回归 |
| utils/ 目录变更 | 对应工具类测试 + 使用方集成测试 | 工具类影响面可控 |
| 依赖升级 | 全量测试 | Qt 版本升级等需完整回归 |

### 8.4 缺陷回归

- 每个已修复的 Bug 必须编写对应的回归测试用例
- 回归用例命名规则：`test_regression_issue{ID}_{description}`
- 回归用例永久保留在测试集中
